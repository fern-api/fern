# Claude Code Automation Recommendations for Fern

Generated: 2026-03-10

---

## Codebase Profile

- **Type**: TypeScript monorepo (pnpm workspaces + Turborepo), with Go, Python, and Java generators
- **Framework**: Custom CLI tool (yargs-based), Docker-containerized code generators
- **Key Libraries**: Biome (lint/format), Prettier (YAML), Vitest (testing), Husky (git hooks), tsup (bundling)
- **Scale**: ~97 workspace packages, ~58 CI workflows, 17 language generators
- **Existing Automations**:
  - PostToolUse hooks: Biome auto-fix on TS/JS/JSON edits, Prettier on YAML edits
  - Slash command: `/fern:seed-debug` for generator debugging
  - Permissions: deny rules for generated code, env files, build artifacts

## MCP Servers

### 1. context7 -- Live Documentation Lookup

**Why**: Fern's generators span 9+ languages (TypeScript, Python, Go, Java, C#, Ruby, PHP, Rust, Swift), each with their own AST libraries and SDK patterns. context7 provides instant access to up-to-date documentation for any library, eliminating guesswork when working across generator codebases.

**Install**:
```bash
claude mcp add context7 -- npx -y @upstash/context7-mcp@latest
```

**Team sharing**: Add to `.mcp.json` so all contributors get it automatically.

### 2. Sequential Thinking -- Structured Reasoning

**Why**: Already referenced in the `/fern:seed-debug` skill. Useful for multi-step debugging of generator output issues, IR migration planning, and reasoning through complex cross-generator changes. The codebase's IR-centric architecture means changes often cascade across multiple packages.

**Install**:
```bash
claude mcp add sequential-thinking -- npx -y @anthropic/mcp-sequential-thinking
```

## Skills

### 1. IR Change Workflow Skill

**Why**: The IR update workflow is the most complex and error-prone process in the repo: update schema definitions, run `pnpm ir:generate`, update `CHANGELOG.md` and `VERSION`, compile, fix breaking `switch` statements across generators, create IR migrations, and coordinate multiple PRs. A skill can encode this entire multi-stage checklist with validation at each step.

**Create**: `.claude/skills/ir-change/SKILL.md`

```yaml
---
name: ir-change
description: "Guide and validate IR schema changes through the full update workflow"
argument-hint: "<description of the IR change>"
---
```

**Invocation**: User-only (via `/ir-change`)

**Suggested content**: Encode the "Feature Addition Workflow" from CLAUDE.md as a step-by-step skill with validation commands at each stage:
1. Schema update in `/packages/ir-sdk/fern/apis/ir-types-latest/definition/`
2. Run `pnpm ir:generate` and verify output
3. Run `pnpm compile` and list all files with new `switch` cases needing updates
4. Guide through `CHANGELOG.md`, `VERSION`, and `versions.yml` updates
5. Validate IR migration compatibility

### 2. Generator Scaffold Skill

**Why**: Creating a new generator or adding a feature to an existing one follows a rigid pattern (implement `AbstractGeneratorCli`, wire up Docker, add seed fixtures, configure CI). A skill can scaffold the boilerplate and remind developers of all the touchpoints.

**Create**: `.claude/skills/new-generator-feature/SKILL.md`

```yaml
---
name: new-generator-feature
description: "Scaffold a new generator feature with proper seed fixtures and CI config"
argument-hint: "<generator-name> <feature-description>"
---
```

**Invocation**: User-only (via `/new-generator-feature`)

## Hooks

### 1. PreToolUse: Block Lock File Edits

**Why**: `pnpm-lock.yaml` is 200K+ lines and should only change via `pnpm install`. Accidental edits to this file waste time and create noisy diffs.

**Where**: `.claude/settings.json` (add to existing `hooks` section)

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|MultiEdit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | { read file_path; if echo \"$file_path\" | grep -qE '(pnpm-lock\\.yaml|yarn\\.lock|package-lock\\.json)$'; then echo 'BLOCK: Do not edit lock files directly. Use pnpm install instead.' >&2; exit 1; fi; }"
          }
        ]
      }
    ]
  }
}
```

### 2. PreToolUse: Block Generated IR SDK Edits (Already Configured)

**Status**: Already in place via deny rules. The existing `Edit(./packages/ir-sdk/src)` and `Edit(./generators/go/internal/fern/ir)` deny rules correctly protect generated code. No changes needed.

**Additional consideration**: Add a PreToolUse hook for `/seed/` fixture output directories. These are regenerated by seed tests and should not be hand-edited:

```json
{
  "matcher": "Edit|MultiEdit|Write",
  "hooks": [
    {
      "type": "command",
      "command": "jq -r '.tool_input.file_path' | { read file_path; if echo \"$file_path\" | grep -qE '/seed/(csharp|go|java|php|python|ruby|rust|swift|ts)-(sdk|model)/'; then echo 'BLOCK: Seed fixture outputs are generated. Run seed tests to update them.' >&2; exit 1; fi; }"
    }
  ]
}
```

## Subagents

### 1. Cross-Generator Impact Analyzer

**Why**: Changes to shared infrastructure (IR SDK, base generator, CLI workspace loader) can break multiple generators silently. A subagent can run in parallel to analyze which generators are affected by a change and flag which seed fixtures need re-testing.

**Create**: `.claude/agents/cross-generator-impact.md`

```markdown
# Cross-Generator Impact Analyzer

You are a code impact analysis agent for the Fern monorepo.

## Task
Given a set of changed files, determine which generators and seed fixtures are affected.

## Process
1. Read the changed files from the git diff
2. Trace import dependencies to identify affected packages
3. For changes in `/packages/ir-sdk/`: flag ALL generators as affected
4. For changes in `/generators/base/`: flag all TypeScript-based generators
5. For changes in a specific generator: identify which seed fixtures test the affected code paths
6. Output a prioritized list of seed test commands to run

## Output Format
- List of affected generators with confidence level
- Specific `pnpm seed test` commands to validate
- Any IR migration concerns
```

### 2. PR Description Generator

**Why**: With 58 CI workflows and strict PR title conventions (`<type>(<scope>): <description>`), a subagent can analyze the diff, determine the correct type/scope, draft a compliant title, and generate a description following the PR template.

**Create**: `.claude/agents/pr-drafter.md`

```markdown
# PR Description Drafter

You are a pull request drafting agent for the Fern monorepo.

## Task
Analyze the current branch's changes and draft a PR title and description.

## Rules
1. PR title must follow: `<type>(<scope>): <description>` (under 70 chars)
2. Allowed types: fix, feat, revert, break, chore
3. Allowed scopes: docs, changelog, internal, cli, typescript, python, java, csharp, go, php, ruby, seed, postman, ci, lint, fastapi, spring, express, openapi, deps, deps-dev, fiber, pydantic, ai-search, swift, rust
4. If changes span multiple scopes, use the primary one
5. Follow the PR template in `.github/pull_request_template.md`

## Process
1. Run `git log main..HEAD` and `git diff main...HEAD --stat`
2. Analyze the nature of changes (new feature, bug fix, refactor, etc.)
3. Determine the correct type and scope
4. Draft title and body
```
