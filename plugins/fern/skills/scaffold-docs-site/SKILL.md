# Scaffold a Fern docs site

Fern docs sites are configured by `fern/docs.yml` plus MDX files in
`fern/pages/`. This skill creates that scaffolding and wires it to any
existing API definition.

## Pre-flight

1. Confirm there is a Fern project in the current directory (`fern.config.json`
   is the marker). If not, invoke the `init-fern-project` skill first.

2. Check for an existing `fern/docs.yml`. If one exists, **stop** and confirm
   before overwriting — the user may have customized navigation, theming, or
   API references.

3. Identify any API definitions referenced in `generators.yml` so the docs
   site can pick up an API Reference section automatically.

## Bootstrap

If the project does not have a docs config yet, create a minimal
`fern/docs.yml` by hand (or let `fern docs dev` generate it interactively
on first run if the CLI version supports it). The safe non-interactive
approach is to write the file directly:

```yaml
instances:
  - url: <org>.docs.buildwithfern.com # replace <org>
navigation:
  - section: Introduction
    contents:
      - page: Welcome
        path: ./pages/welcome.mdx
  - api: API Reference
colors:
  accent-primary:
    light: "#7C3AED"
    dark: "#A78BFA"
```

And a placeholder page at `fern/pages/welcome.mdx`:

```mdx
# Welcome

Documentation generated by Fern.
```

## Migration from another docs tool

If the user is migrating from Mintlify, ReadMe, or similar, ask whether they
want Fern to import existing MDX content. For Mintlify specifically, the
import command exists:

```bash
fern docs migrate --from mintlify --path <path-to-mintlify-docs>
```

If no migration command exists for their source tool, manually copy the MDX
into `fern/pages/` and update `navigation` in `docs.yml`.

## After scaffold

Tell the user:

1. Preview locally: `fern docs dev`
2. Validate: `fern docs check`
3. Publish: `fern docs publish`

For Fern's customization options (colors, fonts, logos, custom domain),
point them to https://buildwithfern.com/learn/docs.
