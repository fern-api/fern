# Customizing your CLI

Your Fern-generated CLI ships with sensible defaults — every operation from your spec is reachable as a subcommand, auth comes from environment variables, output is pretty-printed JSON. For most APIs, that's everything you need.

This guide is for when it isn't:

- You want `acme me` to be a shortcut for `acme users get-current-user`.
- You want every list response unwrapped from a `{"data": ...}` envelope.
- You want to add a `verify-webhook` command that doesn't exist in the spec.
- You want `acme users delete` to succeed even when the user is already gone.
- You want to inject an `X-Audit-CLI: <version>` header on every HTTP request.

All of these are **customizations**. You add them by editing `src/main.rs` directly.

## How customization works

The generator emits `src/main.rs`:

```rust
// src/main.rs — generated by `fern generate`.
use fern_cli_sdk::app::CliApp;
use fern_cli_sdk::auth::ApiKeyAuth;
use fern_cli_sdk::openapi::OpenApiBinding;

fn main() {
    CliApp::new("acme")
        .auth(ApiKeyAuth::new("ApiKey").env("ACME_API_KEY"))
        .binding(
            OpenApiBinding::new()
                .spec(include_str!("../specs/openapi.yml")),
        )
        .run()
}
```

You customize by adding builder calls and writing `fn` items in the same file:

```rust
// src/main.rs — after your edits
use clap::{ArgMatches, Command};
use fern_cli_sdk::app::CliApp;
use fern_cli_sdk::auth::ApiKeyAuth;
use fern_cli_sdk::error::CliError;
use fern_cli_sdk::openapi::{AppContext, OpenApiBinding};
use serde_json::Value;

fn main() {
    CliApp::new("acme")
        .auth(ApiKeyAuth::new("ApiKey").env("ACME_API_KEY"))
        .binding(
            OpenApiBinding::new()
                .spec(include_str!("../specs/openapi.yml")),
        )
        .command_under(
            &["webhooks"],
            verify_cmd(),
            OpenApiBinding::handler(verify_handler),
        )
        .alias(&["users", "get-current-user"], "me")
        .transform_response(&["**", "list"], |v, _path| {
            Ok(v.get("data").cloned().unwrap_or(v))
        })
        .run()
}

fn verify_cmd() -> Command { /* ... */ }
fn verify_handler(args: &ArgMatches, ctx: &AppContext) -> Result<(), CliError> { /* ... */ }
```

When you run `fern generate` again, [Fern Replay](https://buildwithfern.com/learn/sdks/overview/custom-code#replay) reapplies your edits to the freshly-generated `main.rs` via 3-way merge. If the generator and your edits touched the same lines, the regeneration PR shows the conflict and you resolve it with `fern replay resolve`.

For larger projects, split into sibling modules (`src/hooks.rs`, `src/commands/`) and list them in `.fernignore` so the generator never touches them. This guide stays single-file.

## The customization surface

The SDK is deliberately small. Customizations attach via:

| Where | What it does |
|---|---|
| Builder methods on `CliApp` (after `.binding(...)`) | Aliases, deprecation, custom commands, response transforms, error recovery |
| `CliApp::http(\|t\| ...)` / `grpc` / `websocket` | Transport infra (TLS, proxy, retry policy, user-agent), `before_send` hook (not yet implemented) |
| Builder methods on `OpenApiBinding` / `GraphqlBinding` / etc. | Spec, auth wiring, paginators, server vars |

Anything that doesn't fit one of those gets written as a **custom command** — a `fn` item that calls `ctx.invoke_by_name(...)` and does whatever Rust logic the case requires.

## Recipes

### Add an alias for a long command path

```rust
fn main() {
    CliApp::new("acme")
        .binding(/* ... */)
        .alias(&["users", "get-current-user"], "me")
        .run()
}
```

`acme me` now works alongside `acme users get-current-user`.

### Default a flag from an env var

> *Not yet implemented.* `default_arg` and `ArgSource` are spec'd but not yet wired to dispatch. For now, use `server_var` on the binding for global path parameters, or read environment variables directly in a custom command handler.

### Strip an envelope from list responses

```rust
use fern_cli_sdk::reexport::serde_json::Value;

fn main() {
    CliApp::new("acme")
        .binding(/* ... */)
        .transform_response(&["**", "list"], |mut v: Value, _ctx| async move {
            Ok(v.get_mut("data").map(Value::take).unwrap_or(v))
        })
        .run()
}
```

One closure covers every list endpoint. Same shape works for any post-decode transform.

### Add a custom command

Custom commands register on `CliApp`, not on the binding. Use `OpenApiBinding::handler()` (or `GraphqlBinding::handler()`) to wrap a typed handler that receives the protocol-specific `AppContext`:

```rust
use clap::{Arg, ArgMatches, Command};
use fern_cli_sdk::app::CliApp;
use fern_cli_sdk::auth::ApiKeyAuth;
use fern_cli_sdk::error::CliError;
use fern_cli_sdk::openapi::{AppContext, OpenApiBinding};

fn main() {
    CliApp::new("acme")
        .auth(ApiKeyAuth::new("ApiKey").env("ACME_API_KEY"))
        .binding(
            OpenApiBinding::new()
                .spec(include_str!("../specs/openapi.yml")),
        )
        .command_under(
            &["webhooks"],
            verify_cmd(),
            OpenApiBinding::handler(verify_handler),
        )
        .run()
}

fn verify_cmd() -> Command {
    Command::new("verify")
        .about("Verify an Acme webhook signature locally")
        .arg(Arg::new("signature").long("signature").required(true))
        .arg(Arg::new("body").long("body").required(true))
}

fn verify_handler(args: &ArgMatches, _ctx: &AppContext) -> Result<(), CliError> {
    let sig = args.get_one::<String>("signature").unwrap();
    let body = args.get_one::<String>("body").unwrap();
    // ... compute HMAC, compare, print ...
    Ok(())
}
```

The handler receives a real `AppContext` — it can chain API calls via `ctx.invoke_by_name(...)` if needed. Note: `transform_response` / `recover_error` hooks do **not** fire for custom commands.

### Override a spec-generated operation

```rust
fn main() {
    CliApp::new("acme")
        .auth(ApiKeyAuth::new("ApiKey").env("ACME_API_KEY"))
        .binding(
            OpenApiBinding::new()
                .spec(include_str!("../specs/openapi.yml")),
        )
        .replace_command(
            &["users", "get"],
            users_get_cmd(),
            OpenApiBinding::handler(users_get_handler),
        )
        .run()
}
```

`replace_command` fails at startup if `users get` doesn't exist — guaranteed to override something, never silently add. Chain multiple calls for several overrides. *(Note: `replace_command` is spec'd but not yet implemented.)*

### Inject an audit header on every HTTP request

```rust
fn main() {
    CliApp::new("acme")
        .http(|t| {
            t.before_send(&["**"], |b, _| async move {
                Ok(b.header("X-Audit-CLI", env!("CARGO_PKG_VERSION")))
            })
        })
        .binding(/* ... */)
        .run()
}
```

Transport-scope hooks fire across every binding using that transport. For one-binding scope, gate on `ctx.binding_name()` inside the closure.

### Recover from idempotent-delete errors

Goal: `acme users delete --id ...` succeeds (exit 0) even when the user is already gone.

```rust
use fern_cli_sdk::reexport::serde_json::json;

fn main() {
    CliApp::new("acme")
        .binding(/* ... */)
        .recover_error(&["**", "delete"], |err, _ctx| async move {
            if err.status == 404 {
                Ok(Some(json!({ "deleted": true, "id_not_found": true })))
            } else {
                Ok(None)
            }
        })
        .run()
}
```

`Ok(Some(v))` short-circuits with `v` as the response; `Ok(None)` lets the error propagate.

### Load shared config

Use a `static` slot — Rust's stdlib has `OnceLock<T>` for this:

```rust
use std::sync::OnceLock;

#[derive(Clone)]
struct MyConfig { workspace: String, region: String }

static CONFIG: OnceLock<MyConfig> = OnceLock::new();

fn config() -> &'static MyConfig {
    CONFIG.get_or_init(|| MyConfig {
        workspace: std::env::var("ACME_WORKSPACE").unwrap_or_default(),
        region:    std::env::var("ACME_REGION").unwrap_or_else(|_| "us-east-1".into()),
    })
}

async fn my_handler(_args: &ArgMatches, ctx: &AppContext) -> Result<()> {
    let cfg = config();
    // ... use cfg ...
    Ok(())
}
```

Anywhere a handler or closure runs, `config()` returns the same `&'static MyConfig`. For async-loaded config, swap to `tokio::sync::OnceCell`.

### Custom auth scheme (AWS SigV4, OAuth1, HMAC, ...)

Implement `AuthProvider<C>` for the appropriate transport:

```rust
use async_trait::async_trait;
use fern_cli_sdk::{AuthProvider, Operation, Result};
use fern_cli_sdk::transport::http;

pub struct AwsSigV4 { /* access key, secret, region, service */ }

#[async_trait]
impl AuthProvider<http::Credentials> for AwsSigV4 {
    async fn credentials_for(&self, op: &dyn Operation) -> Result<Option<http::Credentials>> {
        // produce 4 headers + 1 query field, return as http::Credentials
        Ok(Some(http::Credentials { /* ... */ }))
    }
    fn has_credentials_for(&self, _op: &dyn Operation) -> bool { true }
}

fn main() {
    CliApp::new("acme")
        .binding(
            OpenApiBinding::new()
                .spec(include_str!("../specs/openapi.yml"))
                .auth_provider("AwsSigV4", std::sync::Arc::new(AwsSigV4::new(/* ... */))),
        )
        .run()
}
```

Same shape for any auth scheme that doesn't fit the typed builders (`BearerAuth`, `ApiKeyAuth`, `BasicAuth`, `OAuth2Auth`) or binding-level `auth_scheme_env`.

### Custom paginator

Implement `Paginator` (for HTTP transport) and register per-operation:

```rust
use async_trait::async_trait;
use fern_cli_sdk::transport::http::{self, Paginator};

pub struct MyCursorPaginator;

#[async_trait]
impl Paginator for MyCursorPaginator {
    async fn next_request(&self, prev_page: &Value, prev_req: &http::Invocation)
        -> Result<Option<http::Invocation>>
    {
        // build the next request from prev_page; None = stop
        Ok(None)
    }
    fn extract_items(&self, page: &Value) -> Vec<Value> {
        page["items"].as_array().cloned().unwrap_or_default()
    }
}

fn main() {
    CliApp::new("acme")
        .auth(ApiKeyAuth::new("ApiKey").env("ACME_API_KEY"))
        .binding(
            OpenApiBinding::new()
                .spec(include_str!("../specs/openapi.yml"))
                .paginator(&["events", "list"], std::sync::Arc::new(MyCursorPaginator)),
        )
        .run()
}
```

## Hook reference

| Method | Where | What |
|---|---|---|
| `alias(path, alias)` | CliApp | clap alias for a command path |
| `hide(path)` | CliApp | Hide a command from `--help` |
| `deprecate(path, msg)` / `stability(path, level)` | CliApp | Maturity tag + `--maturity` gating |
| `auth(builder)` | CliApp | Root-level typed auth (`BearerAuth`, `ApiKeyAuth`, `BasicAuth`, `OAuth2Auth`) |
| `transform_response(path, \|v, ctx\| ...)` | CliApp | Transform a decoded response |
| `recover_error(path, \|err, ctx\| ...)` | CliApp | Convert API error → synthetic success |
| `user_agent` / `ca_bundle_env` / `proxy_env` / `retry_policy` | Transport | Transport infra |
| `default_paginator(Arc::new(p))` | Transport | Default paginator for the transport |
| `before_send(path, \|b, ctx\| ...)` | Transport | Mutate every outbound request |
| `command(cmd, handler)` / `command_under(path, cmd, handler)` | CliApp | Custom commands (use `OpenApiBinding::handler()` for typed access) |
| `replace_command(path, cmd, handler)` | CliApp | Replace a spec-generated command (not yet implemented) |
| `paginator(path, Arc::new(p))` | Binding | Per-operation paginator |
| `auth_provider(scheme, Arc::new(p))` | Binding | Custom auth scheme |

That's the whole customization surface. Anything else is either:
- A custom command (`command_under` + a `fn` item)
- An `AuthProvider` / `Paginator` trait impl
- Code in `fn main()` before `.run()` (lifecycle, config loading, panic handler)

## Splitting into multiple files

For larger projects:

```
src/
├── main.rs       # GENERATED. Builder chain + hook registrations.
├── hooks.rs      # Hook bodies — your code, listed in .fernignore.
└── commands/
    ├── mod.rs
    └── webhooks.rs
```

```
# .fernignore
src/hooks.rs
src/commands/
```

```rust
// src/main.rs
mod hooks;
mod commands;

fn main() {
    CliApp::new("acme")
        .auth(ApiKeyAuth::new("ApiKey").env("ACME_API_KEY"))
        .binding(
            OpenApiBinding::new()
                .spec(include_str!("../specs/openapi.yml")),
        )
        .command_under(
            &["webhooks"],
            commands::webhooks::cmd(),
            OpenApiBinding::handler(commands::webhooks::handle),
        )
        .transform_response(&["**", "list"], hooks::strip_data)
        .run()
}
```

The `mod hooks;` / `mod commands;` lines are customer edits Replay reapplies on regeneration.

## When regeneration changes things

`fern generate` regenerates `main.rs`. Fern Replay reapplies your edits via 3-way merge. Two failure modes worth knowing about:

- **Merge conflicts.** Generator and your edits touched the same lines. The regen PR shows the conflict; run `fern replay resolve` locally to walk through it.
- **Operation-path drift.** If you registered `transform_response(&["users", "list"], ...)` and the spec renames `users/list` to `users/index`, your edit still merges cleanly — but the hook **hard-fails at startup** with `CliError::Validation` listing the unmatched patterns. This catches the problem immediately rather than silently ignoring the hook.

For full-file ownership (helpers, hand-written modules), list the files in `.fernignore`.

## Testing your customizations

Enable the SDK's `test-util` feature:

```toml
[dev-dependencies]
fern-cli-sdk = { version = "...", features = ["test-util"] }
```

Then unit-test hook bodies and handlers in isolation:

```rust
use fern_cli_sdk::AppContext;
use fern_cli_sdk::reexport::serde_json::json;

#[tokio::test]
async fn strip_data_envelope_works() {
    let ctx = AppContext::for_test().build();
    let input = json!({ "data": [{ "id": 1 }, { "id": 2 }] });
    let output = strip_data(input, &ctx).await.unwrap();
    assert_eq!(output, json!([{ "id": 1 }, { "id": 2 }]));
}
```

For end-to-end behavior, use `wiremock` as today's SDK tests do.

## When something doesn't fit

The customization surface above is the supported API. If you need something it doesn't cover — modifying the spec at runtime, building your own transport, etc. — every internal type is `pub` with no stability guarantee. Read the SDK source, use the type directly. If you reach for this escape hatch, file an issue; there's probably a missing capability the SDK should ship.
