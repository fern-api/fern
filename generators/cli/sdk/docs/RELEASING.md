# Releasing

This document covers the release pipeline for maintainers. For how to contribute a change, see [CONTRIBUTING.md](CONTRIBUTING.md).

## Overview

Releases are driven by git tags. cargo-dist builds and publishes from any `vX.Y.Z` tag — there is no version-bump automation in this repo.

1. Bump `version` under `[workspace.package]` in `Cargo.toml` and commit it to `main`. The tag in the next step points at `HEAD`, and cargo-dist requires the committed version to match the tag — an uncommitted bump fails the release.
2. Run `scripts/tag-release.sh` — it reads that version, then creates and pushes the matching `vX.Y.Z` tag (idempotent: skips if the tag already exists; aborts if the working tree has uncommitted changes).
3. The `Release` workflow (`.github/workflows/release.yml`, generated by cargo-dist) picks up the tag, builds binaries for all platform targets, and creates a GitHub Release with generated notes.

The build runs on the repository's built-in `GITHUB_TOKEN` — no secret setup is required for the build-and-release step.

> Note: the pipeline has not been exercised with a real release yet. The first tag push should be treated as a validation run.

## Platform Targets

Each release builds binaries for:

| Target | Platform |
|--------|---------|
| `aarch64-apple-darwin` | macOS Apple Silicon |
| `x86_64-apple-darwin` | macOS Intel |
| `aarch64-unknown-linux-gnu` | Linux ARM64 (glibc) |
| `aarch64-unknown-linux-musl` | Linux ARM64 (musl) |
| `x86_64-unknown-linux-gnu` | Linux x86\_64 (glibc) |
| `x86_64-unknown-linux-musl` | Linux x86\_64 (musl) |
| `x86_64-pc-windows-msvc` | Windows x86\_64 |

Shell, PowerShell, and npm installers are also generated by cargo-dist.

## Enabling Publishing (Not Yet Active)

| What | Status | How to enable |
|------|--------|---------------|
| npm registry | Deferred | Change `publish-jobs = []` → `publish-jobs = ["npm"]` in `dist-workspace.toml` |
| Homebrew | Not configured | Add `"homebrew"` to `installers` and `tap = "fern-api/homebrew-fern"` in `dist-workspace.toml`; create the tap repo on GitHub |
| crates.io | Deferred | Uncomment `publish-crates-io` job in `.github/workflows/release.yml`, add `CARGO_REGISTRY_TOKEN` secret |

Enable these only after validating the end-to-end pipeline with a real release.

## Changesets

`.changeset/*.md` files are kept as per-change notes but are **not** wired into this release flow — nothing consumes them to compute a version or changelog. They remain as raw material for a future autoversioning/changelog improvement.
